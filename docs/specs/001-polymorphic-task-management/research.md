# 연구 (Research)

## 1. 외부 시스템 통합 전략

- **결정**: Task 실행 및 데이터 영속성을 위해 전용 어댑터/플러그인 계층을 통해 외부 시스템(저장소, 실행기)과 통합합니다.
- **근거**: 모듈 간의 낮은 결합도를 유지하고 SOLID 원칙을 준수하며 확장성을 높일 수 있습니다. 새로운 외부 시스템 통합 시 TaskManager의 코드를 수정할 필요가 없습니다.
- **고려된 대안**:
    - TaskManager가 직접 외부 시스템 API 호출: 결합도 증가, 유연성 저하.
    - 메시지 큐를 통한 비동기 통신: 구현 복잡성 증가, 실시간 피드백 어려움.

## 2. 미등록 Task 유형 정의 처리

- **결정**: 미등록 Task 유형 정의 시 `std::runtime_error`와 같은 예외를 발생시킵니다.
- **근거**: 개발자에게 즉각적인 피드백을 제공하고, 잘못된 Task 유형 정의가 시스템에 등록되는 것을 방지하여 시스템의 예측 가능성과 안정성을 보장합니다.
- **고려된 대안**:
    - 오류 코드 반환: 호출자가 명시적으로 확인해야 하며, 오류 처리 누락 위험.
    - 기본 Task 유형으로 대체: 예상치 못한 동작 유발, 예측 가능성 저해.

## 3. Task 정의/실행 요청 동시성 처리

- **결정**: Task 정의/실행 요청 시 내부 뮤텍스/락을 사용하여 동기화합니다.
- **근거**: 데이터 일관성을 보장하고 경쟁 조건을 방지하는 가장 직접적인 방법입니다. 구현이 비교적 간단합니다.
- **고려된 대안**:
    - 요청을 큐에 넣어 순차 처리: 응답성 저하, 실시간성 중요 시 부적합.
    - 낙관적 잠금(Optimistic Locking): 충돌 드물 시 성능 이점, 충돌 시 재시도 로직으로 복잡성 증가.

## 4. Task 관리 모듈 관측성(Observability) 요구사항

- **결정**: 모든 Task 생명주기 이벤트 로깅 및 Task 상태/진행률 메트릭 제공.
- **근거**: Task 관리 모듈의 안정적인 운영을 위한 기본적인 관측성 요구사항을 충족합니다. 문제 발생 시 원인 분석 및 감사 추적에 필수적이며, 시스템 운영 상태 파악 및 성능 병목 현상 식별에 도움이 됩니다.
- **고려된 대안**:
    - 분산 트레이싱 지원: 복잡한 분산 시스템에 유용하나, 초기 구현 단계에서는 복잡성이 높음.

## 5. Task 관리 모듈 보안 및 개인 정보 보호 고려사항

- **결정**: 현재 단계에서는 Task 정의/실행에 대한 접근 제어(Authorization)를 구현하지 않습니다.
- **근거**: 현재 개발 단계의 범위 및 우선순위를 고려하여 결정되었습니다. 향후 시스템의 무결성과 보안을 위해 접근 제어 구현이 필요할 수 있습니다.
- **고려된 대안**:
    - Task 정의/실행에 대한 접근 제어(Authorization) 구현: 시스템 무결성 및 보안 보장, 오용 방지. (현재 단계에서는 제외)
    - Task 데이터 암호화: 민감 정보 보호에 도움, 성능 오버헤드 발생 가능. (현재 단계에서는 제외)
    - 특정 개인 식별 정보(PII) 처리하지 않음: 규정 준수 및 데이터 관리 복잡성 감소. (현재 Task 데이터에 PII 포함되지 않음으로 가정)
